# Use Case: Reject Invalid Input

## 1. Overview
- **Use Case ID:** UC-006
- **Primary Actor:** End User (browser) / API integrator
- **Description:** When a write operation is submitted with invalid data (empty title, title too long, description too long, wrong types, or missing required fields), the API rejects the request with a structured error response and the UI surfaces a clear message. No data is written.
- **Level:** User goal

## 2. Preconditions
- The TodoApp server is running and reachable.
- The user or API client is attempting a `POST /todos` or `PATCH /todos/{id}` operation.

## 3. Trigger
The user submits a todo form (add or edit) with data that violates a validation rule, or an API client sends a request with an invalid body.

## 4. Main Success Scenario
1. The user submits the add or edit form with invalid data (e.g., blank title).
2. The UI sends the write request (`POST /todos` or `PATCH /todos/{id}`) with the invalid payload.
3. The API's zod validation schema evaluates the request body.
4. Validation fails; the service layer is not invoked and no database write occurs.
5. The API responds `400 Bad Request` with a structured JSON body containing `error` and a `details` array listing each offending field and a human-readable message.
6. The UI displays the error message(s) adjacent to the offending field(s); the form remains open for correction.

## 5. Alternate Flows

**5a. Title is empty or whitespace-only**
- `details` contains an entry for `title`: "Title is required and must be 1-255 characters".

**5b. Title exceeds 255 characters**
- `details` contains an entry for `title`: "Title must not exceed 255 characters".

**5c. Description exceeds 2000 characters**
- `details` contains an entry for `description`: "Description must not exceed 2000 characters".

**5d. Malformed JSON body**
- The API cannot parse the body; responds `400 Bad Request` with `{"error": "Invalid JSON body"}`.
- The UI displays a generic error.

**5e. Wrong type for `completed` field on PATCH**
- `completed` is not a boolean; `details` contains an entry for `completed`: "Must be a boolean".

**5f. API integrator (non-UI) client**
- The same `400` response with structured `details` is returned; the integrator handles it programmatically.

## 6. Postconditions
- **Success (rejection succeeds):** No record is created or modified in the database. The API has returned `400` with actionable error details.
- **Failure (validation incorrectly passes):** Invalid data would be persisted - this is the defect case caught by automated tests.
