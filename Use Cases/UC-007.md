# Use Case: Monitor Health

## 1. Overview
- **Use Case ID:** UC-007
- **Primary Actor:** Operator / Orchestrator (e.g., process monitor, container health check, uptime tool)
- **Description:** An operator or automated system polls the health endpoint to confirm that the TodoApp server process is alive and its database connection is functional.
- **Level:** User goal

## 2. Preconditions
- The TodoApp server process has been started.
- The operator/orchestrator has network access to the server's configured port.

## 3. Trigger
An operator or automated health-check probe sends `GET /health` to the server.

## 4. Main Success Scenario
1. The probe sends `GET /health`.
2. The health module executes a trivial database query (`SELECT 1`) against the SQLite file.
3. The query succeeds, confirming DB reachability.
4. The API responds `200 OK` with JSON body `{"status": "ok"}`.
5. The operator/probe records the service as healthy.

## 5. Alternate Flows

**5a. Database file is unreachable or corrupt**
- At step 2, the `SELECT 1` query throws an error.
- The health module catches the error and the API responds `503 Service Unavailable` with body `{"status": "error", "detail": "database unreachable"}`.
- No internal stack trace or sensitive file-path information is included in the response body.
- The operator/probe records the service as unhealthy and may trigger an alert or restart.

**5b. Server process is completely down**
- The TCP connection is refused; the probe records the service as unreachable without receiving any HTTP response.

**5c. Server is up but under load**
- The response may be slow; as long as the DB query succeeds the endpoint still returns `200 OK`.

## 6. Postconditions
- **Success:** The operator has received `200 {"status":"ok"}`, confirming liveness and DB reachability. No database state is modified (read-only probe query).
- **Failure:** The operator has received `503` or a connection error, indicating the service or its DB is degraded. No database state is modified.
