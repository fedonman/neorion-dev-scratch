# Use Case: View Todos

## 1. Overview
- **Use Case ID:** UC-002
- **Primary Actor:** End User (browser)
- **Description:** The user opens the app and sees the full list of existing todo items, each showing its title, optional description, and completion status.
- **Level:** User goal

## 2. Preconditions
- The TodoApp server is running and reachable.
- The SQLite database file exists and is readable (it may contain zero rows).

## 3. Trigger
The user navigates to the TodoApp URL in a browser (initial page load or manual refresh).

## 4. Main Success Scenario
1. The browser loads the static HTML/JS UI served by the Express server.
2. On load, the UI sends `GET /todos`.
3. The repository executes `SELECT * FROM todos ORDER BY created_at ASC` and returns all rows.
4. The API responds `200 OK` with a JSON array of todo objects (each with `id`, `title`, `description`, `completed`, `created_at`, `updated_at`).
5. The UI renders the list, showing each item's title, description (if present), and a completion indicator (checkbox or strikethrough).
6. Completed items are visually distinguished from pending ones.

## 5. Alternate Flows

**5a. No todos exist**
- At step 4, the API returns `200 OK` with an empty array `[]`.
- The UI renders an empty-state message (e.g., "No todos yet - add one above!").

**5b. Server unreachable**
- At step 2, the fetch call fails; the UI displays a network/loading error message.

**5c. Database unreachable**
- The repository query throws; the API returns `503`; the UI displays an error message.

## 6. Postconditions
- **Success:** The UI reflects the current state of all todos in the database. No database state is modified.
- **Failure:** The UI shows an error; no database state is modified.
