# Use Case: Toggle Todo Complete

## 1. Overview
- **Use Case ID:** UC-004
- **Primary Actor:** End User (browser)
- **Description:** The user marks a pending todo as completed, or marks a completed todo back as pending. The change is persisted and reflected immediately in the UI.
- **Level:** User goal

## 2. Preconditions
- The TodoApp server is running and reachable.
- At least one todo item exists in the database and is visible in the UI.

## 3. Trigger
The user clicks the completion checkbox (or toggle control) on a todo item.

## 4. Main Success Scenario
1. The user clicks the checkbox on a todo item.
2. The UI reads the current `completed` value for the item and computes the new value (boolean flip).
3. The UI sends `PATCH /todos/{id}` with JSON body `{"completed": <new boolean>}`.
4. The API validates the request (`completed` must be a boolean; `id` must reference an existing record).
5. The service loads the existing record by `id`.
6. The service sets `completed` to the new value and updates `updated_at`.
7. The repository executes an UPDATE for the record.
8. The API responds `200 OK` with the full updated todo JSON.
9. The UI updates the item's visual state (checkbox checked/unchecked, strikethrough applied or removed).

## 5. Alternate Flows

**5a. Todo not found**
- At step 5, if no record matches `id`, the API returns `404 Not Found`.
- The UI reverts the checkbox to its previous state and displays an error message.

**5b. Invalid payload**
- At step 4, if `completed` is not a boolean, the API returns `400 Bad Request`.
- The UI reverts the checkbox and displays an error.

**5c. Server unreachable**
- At step 3, the fetch call fails; the UI reverts the checkbox and shows a network error.

## 6. Postconditions
- **Success:** The todo record's `completed` flag is updated in the database and `updated_at` is refreshed. The UI reflects the new completion state.
- **Failure:** The database record is unchanged; the UI reverts to the prior visual state.
