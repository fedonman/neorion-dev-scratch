# Use Case: Edit a Todo

## 1. Overview
- **Use Case ID:** UC-003
- **Primary Actor:** End User (browser)
- **Description:** The user modifies the title and/or description of an existing todo item. The change is persisted and reflected in the UI immediately.
- **Level:** User goal

## 2. Preconditions
- The TodoApp server is running and reachable.
- At least one todo item exists in the database and is displayed in the UI.

## 3. Trigger
The user activates the edit control (e.g., clicks an "Edit" button) on an existing todo item.

## 4. Main Success Scenario
1. The UI enters edit mode for the selected item, pre-populating input fields with the current `title` and `description`.
2. The user modifies the title and/or description.
3. The user confirms the edit (e.g., clicks "Save").
4. The UI sends `PATCH /todos/{id}` with a JSON body containing the changed fields (e.g., `{title?, description?}`).
5. The API validates the request: title (if supplied) must be 1-255 characters; description (if supplied) must be 2000 characters or fewer.
6. The service loads the existing record by `id` and verifies it exists.
7. The service applies the supplied field changes and updates `updated_at` to the current timestamp.
8. The repository executes an UPDATE for the record.
9. The API responds `200 OK` with the full updated todo JSON.
10. The UI exits edit mode and displays the updated item in the list.

## 5. Alternate Flows

**5a. Todo not found**
- At step 6, if no record matches the `id`, the API returns `404 Not Found`.
- The UI displays an error ("Item not found") and may refresh the list.

**5b. Invalid title provided**
- At step 5, if the new title is empty or exceeds 255 characters, the API returns `400 Bad Request` with field-level details.
- The UI remains in edit mode and displays the validation error.

**5c. Description too long**
- At step 5, if description exceeds 2000 characters, the API returns `400 Bad Request`.
- The UI remains in edit mode and displays the error.

**5d. User cancels edit**
- At step 3, if the user clicks "Cancel", no request is sent, the UI exits edit mode, and the item is unchanged.

## 6. Postconditions
- **Success:** The todo record in the database reflects the new title and/or description, with an updated `updated_at` timestamp. The UI shows the updated values.
- **Failure:** The database record is unchanged; the UI shows an appropriate error.
